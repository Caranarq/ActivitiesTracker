# 04 — Data Model (Canonical + Local Offline)

This document defines:
1) Canonical data structures stored in Google Sheets (source of truth) for each data source.
2) Required local-only data structures for offline-first operation (cache, pending changes, recents).
3) Stable identifiers and field formats.

## 1. Data Source Topology (DECIDED)

### 1.1 Activity Sheet
- Exactly ONE active Activity Sheet Google Spreadsheet is used at a time.
- This Spreadsheet is canonical for:
  - `activity_segments`
  - shared configuration tabs such as `categories` and `view_configs` (see section 3)

### 1.2 Events
- Events can have MULTIPLE sources loaded concurrently.
- Each Event source is independent and is cached/synced independently.
- V1 supports Event sources of type:
  - `google_sheets_events` (Google Spreadsheet containing `events` + related tabs)
- Future (not V1, but designed for):
  - `google_calendar` (live calendar data source)

## 2. Global Conventions

### 2.1 Identifiers
- All primary records MUST have a stable `id` generated by the client.
- `id` format: UUID v4 string.

### 2.2 Timestamps and Timezones
- All timestamps stored as ISO 8601 datetime strings, e.g. `2026-03-02T22:15:00`
- All time-capturing records include:
  - `tz_capture`: IANA timezone string, e.g. `America/Monterrey`

### 2.3 Soft Deletes
Canonical rows support soft delete:
- `is_deleted` boolean
- `deleted_at` timestamp (nullable)

### 2.4 Updated-at for Sync
All canonical rows include:
- `updated_at` (ISO 8601 datetime string)
The client updates `updated_at` on every mutation.

### 2.5 Extensible Metadata
- Canonical records MAY include `meta_json` as a JSON string.
- Must be valid JSON if present.

## 3. Canonical Tabs (Google Sheets)

Important: Because sources are independent, the same tab schemas can exist in multiple spreadsheets.
In V1:
- The Activity Sheet spreadsheet MUST contain the shared configuration tabs.
- Each Google-Sheets-based Event source spreadsheet MAY also contain configuration tabs, but the canonical “shared” configuration lives in Activity Sheet by default (see 3.6).

### 3.1 `categories` (Shared Taxonomy) — DECIDED (C1)
Shared categories/labels used by both events and activity segments.

| ID | Column | Type | Required | Notes |
|----|--------|------|----------|------|
| C1 | id | string (UUID) | yes | Primary key |
| C2 | scope | enum | yes | `both` in V1 |
| C3 | name | string | yes | Display name |
| C4 | color_hex | string | yes | `#RRGGBB` |
| C5 | sort_order | number | no | Optional ordering |
| C6 | is_deleted | boolean | yes | Default false |
| C7 | deleted_at | datetime string | no | Nullable |
| C8 | updated_at | datetime string | yes | ISO 8601 |
| C9 | meta_json | string | no | JSON string |

### 3.2 `events`
Stored in each Google-Sheets-based Event source.

| ID | Column | Type | Required | Notes |
|----|--------|------|----------|------|
| E1 | id | string (UUID) | yes | Primary key |
| E2 | source_id | string (UUID) | yes | Local-defined source identifier for traceability |
| E3 | title | string | yes | Human-friendly |
| E4 | category_id | string (UUID) | yes | FK -> categories.id |
| E5 | time_type | enum | yes | `duration`, `instant`, `open_ended` |
| E6 | start_at | datetime string | yes | ISO 8601 |
| E7 | end_at | datetime string | no | Required if `duration`; null for `instant`/`open_ended` |
| E8 | tz_capture | string | yes | IANA TZ |
| E9 | tag | string | no | Optional tag for correlation with daily-cycle |
| E10 | notes | string | no | Free text |
| E11 | recurrence_id | string (UUID) | no | FK -> recurrences.id |
| E12 | is_deleted | boolean | yes | Default false |
| E13 | deleted_at | datetime string | no | Nullable |
| E14 | updated_at | datetime string | yes | ISO 8601 |
| E15 | meta_json | string | no | JSON string |

Validation:
- `duration`: `end_at` required and `end_at >= start_at`
- `instant`: `end_at` must be null
- `open_ended`: `end_at` null until closed

### 3.3 `recurrences` — DECIDED (R1a)
Stored in each Event source that supports recurrence.

| ID | Column | Type | Required | Notes |
|----|--------|------|----------|------|
| R1 | id | string (UUID) | yes | Primary key |
| R2 | rrule | string | yes | RRULE-like string (iCalendar subset) |
| R3 | start_anchor_at | datetime string | yes | Base anchor for rule interpretation |
| R4 | tz_capture | string | yes | IANA TZ used for rule |
| R5 | is_deleted | boolean | yes | Default false |
| R6 | deleted_at | datetime string | no | Nullable |
| R7 | updated_at | datetime string | yes | ISO 8601 |

Required V1 recurrence capabilities:
- Monthly: nth weekday of month
- Monthly: last day of month
- Weekly interval

### 3.4 `activity_segments` — DECIDED (D2)
Stored in the Activity Sheet spreadsheet.

Rule D2:
- Canonical `activity_segments` rows MUST NOT cross midnight boundaries.
- If user enters a segment crossing midnight, the client MUST split it into two rows at 00:00 local time.

| ID | Column | Type | Required | Notes |
|----|--------|------|----------|------|
| A1 | id | string (UUID) | yes | Primary key |
| A2 | date_key | string | yes | `YYYY-MM-DD` (local date for grouping) |
| A3 | start_at | datetime string | yes | ISO 8601 |
| A4 | end_at | datetime string | yes | ISO 8601 |
| A5 | tz_capture | string | yes | IANA TZ |
| A6 | category_id | string (UUID) | yes | FK -> categories.id |
| A7 | linked_tag | string | no | Optional manual correlation to `events.tag` |
| A8 | notes | string | no | Free text |
| A9 | is_deleted | boolean | yes | Default false |
| A10 | deleted_at | datetime string | no | Nullable |
| A11 | updated_at | datetime string | yes | ISO 8601 |
| A12 | meta_json | string | no | JSON string |

Validation:
- `start_at < end_at`
- `date_key` must match local date of `start_at` in `tz_capture` after applying D2 split.

### 3.5 `view_configs`
Stored canonically in Activity Sheet spreadsheet (single user).

| ID | Column | Type | Required | Notes |
|----|--------|------|----------|------|
| V1 | id | string (UUID) | yes | Primary key |
| V2 | view_type | enum | yes | `events_timeline`, `daily_cycles` |
| V3 | name | string | yes | Display name |
| V4 | config_json | string | yes | JSON string (schema defined in UI States) |
| V5 | is_deleted | boolean | yes | Default false |
| V6 | deleted_at | datetime string | no | Nullable |
| V7 | updated_at | datetime string | yes | ISO 8601 |

### 3.6 Configuration Canonicality Rule (V1)
- The Activity Sheet spreadsheet is the canonical home for:
  - `categories`
  - `view_configs`
- Event sources reference `categories.id` values from that canonical taxonomy.
- In V1, category definitions are NOT duplicated per event source.

Implication:
- If user is offline, categories must be available from local cache.

## 4. Local-Only Data (Offline-first)

Implementation storage is flexible, but must support the following logical structures.

### 4.1 Source Registry
Tracks all configured sources.

- `source_id` (UUID)
- `source_type` enum:
  - `activity_sheet`
  - `google_sheets_events`
  - future: `google_calendar`
- `display_name`
- Connection pointer(s):
  - `spreadsheet_id` and/or URL
- `is_active` boolean (only one `activity_sheet` active; multiple event sources active allowed)
- `last_opened_at`

### 4.2 Local Cache (per source, independent)
For each source:
- Cached rows for each canonical tab present in that source
- `last_sync_at` per tab
- `etag`/revision markers if available (details in API contract)

### 4.3 Pending Changes Queue (per source)
Each queued item:
- `queue_id` (UUID)
- `source_id`
- `tab_name`
- `operation` enum: `insert`, `update`, `soft_delete`
- `record_id` (UUID)
- `payload_json` (intended canonical row state)
- `created_at`
- `status` enum: `pending`, `applied`, `blocked_conflict`, `failed`
- `error_message` (nullable)

Ordering:
- Preserve creation order within each `source_id`.
- Sync runs independently per source; UI may present combined status.

### 4.4 Recent Datasets / Recents UI
Recent entries correspond to sources and combinations:
- Must show the active Activity Sheet source
- Must show recently used Event sources
- Must indicate offline availability (cache present)

Minimum per entry:
- `source_id`
- `display_name`
- `last_opened_at`
- `has_offline_cache`

## 5. Conflict Detection (V1, high-level)
A conflict exists when:
- Remote row `updated_at` is newer than the local baseline (captured at last sync)
AND
- A pending local mutation exists for the same `record_id`

Resolution is interactive (ask user), as specified in Happy Path Flows.
